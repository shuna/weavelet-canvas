# ブランチエディタ機能レビュー

## 現状の仕様整理

| 操作 | 挙動 |
|------|------|
| **生成（append）** | activePath 末尾に新ノードを追加。既存ツリーは不変 |
| **ブランチ作成＋生成** | 対象ノードの兄弟として新ノードを作成し、activePath をそこで切り替え。元の下位ノードはツリー上残るが、新分岐側にはコピーされない |
| **合流（merge）** | なし。グラフは常にツリー（DAG ではない） |

補足：コピー＆ペースト（`copyBranchSequence` / `pasteBranchSequence`）は存在するが、ブランチ作成時に自動的に下位を引き継ぐ動作ではなく明示的な手動操作。

---

## 検討結果

### 1. 十分と言える点

- **分岐の作成・切替・削除・比較（diff）**: 基本的な分岐操作は一通り揃っている
- **任意位置での編集・挿入・削除**: mid-chat 操作があり、途中ノードの書き換えが可能
- **合流がないこと自体は妥当**: 会話ツリーでは前の文脈に依存して生成されるため、2つの分岐を機械的にマージすると文脈の整合性が壊れる。Git のような汎用マージは不要

### 2. 機能的に不足している可能性がある操作

#### (a) 「分岐するが下位を引き継ぐ」操作（fork-with-descendants）

**ユースケース**: メッセージ #3 だけ書き換えて、#4〜#10 はそのまま試したい

現状はブランチ作成で下位が切り捨てられるため、手動でコピー＆ペーストする必要がある。ただし根本的なトレードオフが存在する：

- **コピーが意味をなすケース**: ユーザーメッセージの微修正（typo 修正など）で、後続の流れを保持したい場合
- **コピーが危険なケース**: アシスタント応答は前の文脈に依存して生成されているため、途中を変えて下位をそのまま持つと文脈的に不整合な会話になる

**提案**: あると便利だが、「コピーした下位は文脈的に古い」という警告や、再生成の導線とセットで設計する必要がある。

#### (b) リベース（rebase）— ある分岐の基点を別のノードに付け替える

**ユースケース**: 「この分岐の議論、実はあっちの分岐の続きとしてやりたかった」

現状はコピー＆ペースト→元を削除で疑似的に可能だが、専用操作はない。使用頻度を考えると優先度は低いが、パワーユーザー向けには検討の余地がある。

#### (c) ノードの並べ替え（sibling reorder）

同じ親を持つ兄弟ノード間の表示順を変える操作。現状は `createdAt` 順で固定。ブランチが増えた際に「よく使う分岐を左に持ってくる」等の整理ができると便利。

### 3. なくて問題ない操作

| 操作 | 不要な理由 |
|------|-----------|
| **マージ（merge）** | 会話は時系列・文脈依存であり、2分岐の機械的結合は意味をなさない |
| **チェリーピック** | コピー＆ペーストで実質カバーされている |
| **コンフリクト解決** | マージがなければ不要 |

## ボタン文言の改善提案

### 全ボタンの実際の挙動（コード確認済み）

`EditViewButtons.tsx` と各ハンドラの実装から、会話途中でメッセージを編集した際に表示される全6ボタンの挙動:

| # | 現在のラベル | ハンドラ | 当該メッセージ | その先（下位） |
|---|-------------|---------|--------------|--------------|
| 1 | 生成 | `handleGenerateNextOnly` | 上書き | 次の1件だけ削除＆再生成。残りは不変 |
| 2 | 以下全て生成 | `handleGenerate` (modal経由) | 上書き | 以降を全て削除＆再生成 |
| 3 | ブランチ+生成 | `handleBranchGenerate` | 兄弟ノード作成（編集内容） | 続きを生成（元の下位は引き継がない） |
| 4 | ブランチ | `handleBranchOnly` | 兄弟ノード作成（編集内容） | なし（行き止まり） |
| 5 | 保存 | `handleSave` | 上書き | 変更なし |
| 6 | キャンセル | `handleCancel` | 編集破棄 | — |

### 分岐の内部動作

`createBranchState`（`branch-domain.ts:160-193`）では:

```
parentId: fromNode.parentId   ← 分岐元ノードの「親」を共有して兄弟ノードを作成
```

メッセージ #3 を編集して「ブランチ」を押した場合:

```
#2 (親)
├── #3  (元のノード — 元の内容のまま残る)
└── #3' (新ノード — 編集後の内容、続きなし)
```

当該バブル自体が置き換わるのではなく、**#3 の兄弟として新ノードが作られる**。実質的に「一つ上（#2）から分岐している」のが正確な動作。

### 現状ラベルの問題点

1. **「生成」が文脈で意味が変わる** — 途中では「次の1件だけ再生成」、末尾では「続きを生成」。同じラベルで挙動が異なる
2. **「ブランチ」が何を意味するか不明** — 分岐が作られることはわかるが、続きがないことが伝わらない
3. **「保存」と「ブランチ」の対比が弱い** — 「保存＝上書き」「ブランチ＝別の枝に保存」という対を成しているのに、ラベルからそれが読み取れない
4. **上書き系と分岐系の軸が文言に表れていない** — 1,2,5 は上書き系、3,4 は分岐系だが、ラベルからグループが見えない

### 改善案：2軸を明示する統一命名

**軸1: 保存先** — 上書き（現在のノードを変更） or 分岐（新しい兄弟ノードを作成）
**軸2: 生成** — なし / 次のみ / 以下全て / 続きを生成

上書き系は動作（「再生成」「保存」）をそのまま述べる。分岐系は「分岐して〜」を接頭辞にすることでグループを明示する。

| 現在 | 改善案 | 意味 |
|------|--------|------|
| 生成 | **次を再生成** | 上書きし、次の応答1件だけ再生成する |
| 以下全て生成 | **下を全削除して生成** | 上書きし、以降を全て削除して再生成する |
| ブランチ+生成 | **分岐して生成** | 新しい分岐に保存し、続きを生成する |
| ブランチ | **分岐して保存** | 新しい分岐に保存する（続きなし） |
| 保存 | — | 変更なし |
| キャンセル | — | 変更なし |

設計意図:

- **上書き系 (1,2,5)**: 主語なし＝「今のメッセージを上書きする」が暗黙の前提。動作の違い（再生成のスコープ）だけを述べる
- **分岐系 (3,4)**: 「分岐して」が接頭辞として付き、上書きではないことが一目でわかる
- **「分岐して保存」と「保存」の対**: 同じ「保存」という語で終わることで、両者が「生成はしない」操作であることが揃う。違いは「分岐して」の有無＝保存先の違いだけ
- **「分岐して生成」と「次を再生成」「下を全削除して生成」の対**: 全て「生成」で終わるが、接頭辞で保存先とスコープが異なることが伝わる

### ツールチップ

| ボタン | ツールチップ |
|--------|------------|
| 次を再生成 | このメッセージを上書きし、直後の応答のみ再生成します。それ以降は変更されません |
| 下を全削除して生成 | このメッセージを上書きし、以降を全て削除して生成し直します |
| 分岐して生成 | このメッセージの内容で新しい分岐を作り、続きを生成します。元の分岐はそのまま残りますが、以降の会話は引き継がれません |
| 分岐して保存 | このメッセージの内容で新しい分岐を作ります。元の分岐はそのまま残りますが、以降の会話は引き継がれません。続きの生成は行いません |

### UIレイアウトの改善：分岐操作の分離

#### 現状の問題

編集モードのボタンが横一列に6つ並んでおり、上書き系と分岐系が混在している:

```
[生成] [以下全て生成] [ブランチ+生成] [ブランチ] [保存] [キャンセル]
```

#### 改善案：分岐系を BranchSwitcher のスプリットボタンに移動

分岐操作を編集モードから切り離し、閲覧モードの BranchSwitcher（`< 1/3 >`）に統合する。
メッセージ間の「+」ボタン（`NewMessageButton`）をスプリットボタン化し、ドロップダウンで分岐操作を提供する。

```
閲覧モード（各メッセージ間）:

              ┌──────────┐
              │  +  │ ▾  │    ← 楕円のスプリットボタン
              └──────────┘
                      │
                      ├─ 分岐して生成
                      └─ 分岐して保存
```

- 「+」クリック → 従来通り空メッセージ挿入（挙動変更なし）
- 「▾」クリック → ドロップダウンメニューで分岐系操作を選択

```
編集モード（改善後）:

[次を再生成] [下を全削除して生成]          [保存] [キャンセル]
```

#### 利点

1. **編集モードがシンプルになる** — 上書き系4ボタンのみ。横幅に余裕が生まれる
2. **分岐の文脈が自然** — `< 1/3 >` の隣で「+▾」を押せば「4つ目の分岐を作る」と直感的にわかる
3. **編集モードに入らなくても分岐できる** — 現状は編集モードに入らないと分岐ボタンが出ないが、スプリットボタンなら閲覧モードから直接操作可能
4. **既存の「+」の挙動は変わらない** — 破壊的変更なし

#### ドロップダウンメニュー項目の挙動

編集モード外から呼ばれるため、編集中のテキストがない点を考慮:

| メニュー項目 | 挙動 |
|-------------|------|
| **分岐して生成** | 現在のメッセージ内容で兄弟ノードを作成し、続きを生成する |
| **分岐して保存** | 現在のメッセージ内容で兄弟ノードを作成し、編集モードで開く（内容を変えてから保存可能） |

#### 実装への影響

- `NewMessageButton.tsx` をスプリットボタン化（「+」部と「▾」部を一体の楕円ボタンとして構成）
- ドロップダウンコンポーネントの追加（2項目のみのシンプルなメニュー）
- `EditViewButtons.tsx` から `handleBranchOnly` / `handleBranchGenerate` ボタンを除去
- 翻訳ファイルにメニュー項目のラベルを追加
- ロジック自体（`createBranchState` 等）は変更不要

## 総合評価

現状の設計は会話型分岐エディタとして概ね妥当。「分岐は常に発散、合流なし」というモデルは、会話の文脈依存性を考えると正しい制約。

最も検討価値がある追加機能は **(a) fork-with-descendants**。頻出するユースケース（途中の1メッセージだけ変えて再実行したい）に対して、現状はコピペという手動ステップが必要で摩擦がある。ただし文脈不整合のリスクと表裏一体なので、UI 上の警告や「コピー後に下位を一括再生成」といった補助機能とセットで設計するのが望ましい。

UI については以下の2点の改善を提案:

1. **ボタン文言の統一** — 「分岐して〜」接頭辞で上書き系/分岐系を明示。翻訳ファイルの変更のみで対応可能
2. **分岐操作のスプリットボタン化** — 編集モードから分岐ボタンを除去し、メッセージ間の「+」ボタンをスプリットボタンに拡張。編集モードの簡素化と、閲覧モードからの直接分岐を両立
